# Design Direction Reference

Use this when creating a new page, refreshing a visual surface, polishing a hero, or turning a functional layout into a distinctive interface.

## Direction Before Details

- Choose a specific creative direction before touching CSS: editorial lab notebook, Android device console, model observability cockpit, technical magazine, package registry, or conversation workbench.
- Translate that direction into concrete choices for grid, density, materials, color accents, typography, motion, and iconography.
- Make every section earn a role: orient, prove, compare, operate, read, decide, or act. Avoid repeated decorative cards with the same visual weight.
- Preserve the repository's AI x Android identity, but do not make every page a dark violet gradient. Use page-specific accents and quieter structure when the task is operational.

## Composition Rules

- Keep the first viewport purposeful. Home and editorial pages can use expressive composition; catalog and tool pages should show real controls or scannable data immediately.
- Use asymmetry only when it helps hierarchy. Align recurring metadata, controls, chips, and actions on predictable tracks.
- Give each page family a recognizable rhythm:
  - Home: identity, signal, featured systems, article/project paths.
  - Blog/article: reading width, strong headings, pullouts only when they clarify the argument.
  - Catalog: toolbar, filters, counts, dense item grids, stable card dimensions.
  - Chat/feedback: input-first tool surfaces, clear status, trust and recovery states.
- Avoid nested cards, empty bento grids, ornamental statistics, and decorative UI panels that do not contain real content.

## Typography And Copy

- Use type scale to express hierarchy, not spectacle. Hero-scale type belongs only on true hero surfaces.
- Keep headings compact and specific. Prefer concrete labels such as "Model Catalog", "Android Package Builds", or "AI x Android Notes" over vague slogans.
- Long Chinese and English strings must wrap cleanly. Test both languages when adding or moving text.
- Use tabular numerals for aligned counts, scores, dates, stars, package versions, and model metrics.

## Color, Material, And Media

- Start from existing CSS variables, then add semantic tokens only when a new surface role is needed.
- Pair blue/violet with neutral, graphite, Android green, amber, or signal-like accents when a page needs stronger identity.
- Use gradients with restraint. Prefer layered surfaces, borders, shadows, data lines, device-like frames, or real media over generic glow backgrounds.
- If generated or searched imagery is introduced, it must reveal the subject: app screenshots, device context, article topic, model/workflow state, or concrete product detail.

## Anti-Generic Checks

Before finishing a visual change, reject the design if it has:

- A hero that could fit any AI startup after only changing the title.
- Repeated cards with equal weight and no meaningful hierarchy.
- Decorative metrics with no source or decision value.
- Purple/blue-only color treatment across every section.
- Motion that exists only to make static content feel busy.
- Copy that explains the UI instead of labeling real actions and content.
